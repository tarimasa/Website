## はじめに

「シークレットが切れて本番が落ちた」——Entra IDを使ったシステム開発・運用でよく聞くトラブルの代表格です。アプリ登録とクライアントシークレットの仕組みを正しく理解し、ベストプラクティスに沿った管理体制を整えましょう。

## 3つの概念を整理する

### アプリ登録（App Registration）

Entra IDにアプリを「登録」する操作。アプリのアイデンティティを定義するもの。

- アプリケーションID（Client ID）が発行される
- APIのアクセス許可（スコープ）を設定する
- 認証方式（シークレット / 証明書 / フェデレーション）を設定する

### サービスプリンシパル（Service Principal）

アプリ登録から生成される「テナント内でのアプリの実体」。

```
アプリ登録 （グローバルな定義）
    ↓ テナントに追加すると
サービスプリンシパル （テナント内での実体）
    ↓
RBACロール、条件付きアクセス等の対象になる
```

### クライアントシークレット

アプリがEntra IDに「自分はこのアプリです」と証明するためのパスワード。

## クライアントシークレット管理のベストプラクティス

### 1. 有効期限を管理する

| 設定 | リスク |
|------|--------|
| 最大2年 | 切れると突然認証不能になる |
| 推奨 | 6〜12ヶ月で設定し、カレンダーリマインダー必須 |

### 2. シークレットよりも証明書を使う

```
シークレット（文字列）< 証明書（公開鍵認証） ← より安全
```

### 3. 可能ならマネージドIDに移行する

AzureリソースからAzureリソースを呼ぶ場合は**マネージドID**を使えばシークレット不要。

```
【シークレット方式】
GitHub Actions → Entra IDにシークレットで認証 → Blob Storageにアクセス

【マネージドID方式（Azure VM / App Service等）】
Azure VM → Entra IDがマネージドIDを自動管理 → Blob Storageにアクセス
（シークレット不要・ローテーション自動）
```

### 4. フェデレーション資格情報を活用する

GitHub ActionsからAzureにアクセスする場合は、シークレット不要の**OIDC連携（ワークロードIDフェデレーション）**が利用可能。

```yaml
# GitHub Actions でシークレットなしでAzureにログイン
- uses: azure/login@v1
  with:
    client-id: ${{ secrets.AZURE_CLIENT_ID }}
    tenant-id: ${{ secrets.AZURE_TENANT_ID }}
    subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
```

## 最小権限の原則を守る

```
悪い例: Contributor ロールを付与
良い例: Storage Blob Data Reader ロールのみ付与
```

API許可も同様に、必要最小限のスコープだけを付与しましょう。

## よくあるハマりポイント

- **「管理者の同意」を付与し忘れてAPIコールが403になる**
- **複数のシークレットを発行して、どれが使われているか分からなくなる**
- **シークレットをコードにベタ書きしてGitHubに公開してしまう**（Azure Key Vaultを使いましょう）

## まとめ

クライアントシークレットは便利ですが管理コストが高いです。新規開発ではマネージドIDやワークロードIDフェデレーションを積極的に採用し、シークレットレス構成を目指しましょう。既存のシークレット管理は有効期限のアラートと定期ローテーションを仕組み化することが重要です。
