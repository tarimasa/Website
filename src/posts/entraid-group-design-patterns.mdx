## はじめに

「Entra IDのグループが増えすぎて誰も管理できなくなった」——組織のM365環境でよく聞くあるあるです。グループ設計を最初に間違えると後から整理するのが大変です。グループの種類と用途を理解し、シンプルで管理しやすい設計を目指しましょう。

## Entra IDのグループの種類

### 1. セキュリティグループ（静的）

最も基本的なグループ。管理者が手動でメンバーを追加・削除。

- RBACロール割り当て
- 条件付きアクセスの対象
- アプリへのアクセス制御

### 2. 動的セキュリティグループ

ユーザー属性（部署・役職・場所など）に基づいてメンバーが自動更新される。

```
【動的メンバーシップルールの例】
user.department -eq "営業部"
user.jobTitle -contains "Manager"
user.country -eq "JP"
```

**メリット**: 異動・入社・退職時に自動的にグループが更新される
**デメリット**: Entra ID P1ライセンスが必要、ルール評価に数分〜数十分かかる場合がある

### 3. Microsoft 365グループ（M365グループ）

TeamsやSharePoint、Exchangeグループメールボックスと連動するグループ。

- Teamsチームを作成すると自動的に作られる
- 共有メールボックス・SharePointサイトが紐づく
- セキュリティ用途には向かない（外部ゲストが入れる場合があるため）

## 用途別の使い分け

| 用途 | 推奨グループ種類 |
|------|---------------|
| ライセンス割り当て | 動的セキュリティグループ |
| 条件付きアクセスの対象 | セキュリティグループ（静的 or 動的） |
| アプリのアクセス制御 | セキュリティグループ |
| Teams/SharePoint連携 | M365グループ |
| RBAC（Azureリソース） | セキュリティグループ |

## グループ命名規則の重要性

管理しやすくするために命名規則を統一しましょう。

```
【命名規則の例】
sec-[用途]-[対象]
  sec-license-m365bp       → M365 Business Premiumライセンス付与
  sec-ca-exclude-admin     → 条件付きアクセス除外（管理者）
  sec-app-salesforce-user  → Salesforce利用者

dyn-[属性]-[値]
  dyn-dept-sales           → 営業部（動的）
  dyn-location-osaka       → 大阪拠点（動的）
```

## ライフサイクル管理

グループも「作りっぱなし」になりがちです。

- **Entra IDグループの有効期限ポリシー**: 一定期間使われないグループを自動削除
- **アクセスレビュー**: 定期的にグループメンバーの棚卸し
- **グループオーナーの設定**: 管理責任者を必ず設定する

## よくあるハマりポイント

- **動的グループはネスト（グループ内グループ）ができない**
- **M365グループをセキュリティ用途に使うと外部ゲストが意図せずアクセスできる**
- **ライセンス割り当てに使うグループをネストしようとして失敗する**（直接メンバーのみが対象）

## まとめ

グループ設計は「静的 vs 動的」「セキュリティ vs M365」の使い分けが肝心です。ライセンス管理には動的グループ、セキュリティポリシーには静的セキュリティグループを使い、命名規則と有効期限ポリシーで管理不能な増殖を防ぎましょう。
