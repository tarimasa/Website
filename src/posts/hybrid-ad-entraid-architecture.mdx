## はじめに

「社内にオンプレのActive Directoryがあるが、Microsoft 365も使っている」という環境は多くの企業で一般的です。この状況でのID管理をどう整理するか——オンプレADとEntra IDをどう連携させるかを理解することが、ハイブリッドID構成の第一歩です。

## ハイブリッドID構成の全体像

```
【オンプレ側】                    【クラウド側】
Active Directory  ←→  Entra Connect  ←→  Microsoft Entra ID
（社内ドメイン）         （同期エンジン）          （クラウドID）
    ↑                                              ↑
  社内PC                                    Microsoft 365
  オンプレアプリ                             SaaSアプリ
```

Entra Connect（旧 Azure AD Connect）がオンプレADとEntra IDの橋渡しをします。

## 認証方式の3パターン

### 1. パスワードハッシュ同期（PHS）

オンプレADのパスワードハッシュをEntra IDに同期する方式。

| 項目 | 内容 |
|------|------|
| 認証場所 | Entra ID（クラウド）で処理 |
| オンプレ依存 | 低い（同期が止まっても認証継続可能） |
| 実装難易度 | 低い |
| 推奨度 | **Microsoftが最も推奨** |

### 2. パススルー認証（PTA）

認証リクエストをオンプレADに転送して検証する方式。

| 項目 | 内容 |
|------|------|
| 認証場所 | オンプレADで処理 |
| オンプレ依存 | 高い（ADが落ちると認証不可） |
| パスワードをクラウドに置かない | ○ |
| 実装難易度 | 中程度 |

### 3. フェデレーション（AD FS）

AD FSを経由してSAMLで認証する方式。

| 項目 | 内容 |
|------|------|
| 認証場所 | AD FSサーバーで処理 |
| 柔軟性 | 高い（カスタム認証ルール可能） |
| インフラコスト | 高い（AD FSサーバー群が必要） |
| 実装難易度 | 高い |

## どれを選ぶべきか

```
複雑な認証要件がある（スマートカード等）?
  → Yes → フェデレーション
  → No  ↓
パスワードをクラウドに置きたくない?
  → Yes → パススルー認証
  → No  → パスワードハッシュ同期（推奨）
```

特別な要件がなければ**パスワードハッシュ同期 + シームレスSSO**の組み合わせがシンプルで堅牢です。

## Entra ConnectのステージングモードとAADC v2

- **ステージングモード**: 同期設定を変更する際のテスト環境として2台目のサーバーを用意できる
- **Microsoft Entra Connect Sync（v2）**: 最新版。旧来のAzure AD Connect v1は2024年にサポート終了

## よくあるハマりポイント

- **UPNサフィックスの不一致**: オンプレのUPN（user@local.domain）とEntraIDのドメインが一致しない
- **同期除外OU設定のミス**: 同期したくないOUが同期されてしまう
- **パスワードライトバックの設定漏れ**: クラウド側でパスワードリセットしてもオンプレに反映されない

## まとめ

ハイブリッドID構成の核心はEntra Connectによる同期方式の選択です。新規導入であれば迷わずパスワードハッシュ同期を選ぶのが現実解です。既存環境のAD FSからの移行も、Microsoftが移行ウィザードを提供しているため、ぜひ検討してみてください。
