## はじめに

「ゼロトラストを導入したい」と言われたとき、最初の具体的な一手として有効なのが条件付きアクセスポリシーです。VPNや社内ネットワークに頼らず、「誰が・どのデバイスで・どこから・何にアクセスするか」を毎回評価するのがゼロトラストの核心であり、Entra IDの条件付きアクセスはその実装手段になります。

## 条件付きアクセスとは

アクセス要求に対して「条件」を評価し、「許可 / MFA要求 / ブロック」を判断する仕組みです。

```
【評価フロー】
ユーザーがアプリにアクセス
  ↓
[シグナル評価]
  ・ユーザー / グループ
  ・デバイスのコンプライアンス状態（Intune連携）
  ・場所（IPアドレス・国）
  ・アプリ（対象サービス）
  ・リスクレベル（Identity Protection連携）
  ↓
[判定]
  → 許可 / MFA要求 / 準拠デバイス要求 / ブロック
```

## 典型的な設計パターン

### パターン1：全ユーザーにMFAを要求する（基本）

最もシンプルで効果的な出発点。

- **対象**: 全ユーザー
- **条件**: 全クラウドアプリ
- **制御**: MFAを要求

### パターン2：管理者アカウントへの強制MFA

管理者ロールを持つアカウントは常にMFA必須にする。

- **対象**: グローバル管理者・特権ロール保有者
- **条件**: 全アプリ・全場所
- **制御**: MFA必須 + 準拠デバイスのみ許可

### パターン3：海外・匿名IPからのアクセスブロック

国内業務が前提の組織に有効。

- **対象**: 全ユーザー
- **条件**: 許可済みIPレンジ外
- **制御**: ブロック（または管理者のみ除外）

### パターン4：未登録デバイスはWebブラウザのみ許可

個人端末（BYOD）のコントロール。

- **対象**: 全ユーザー
- **条件**: 準拠デバイス以外
- **制御**: アプリ保護ポリシー適用 / ブラウザセッションのみ許可

## 段階的ロールアウト戦略

一気に全員に適用するとヘルプデスクが大変なことになります。

```
フェーズ1: レポートのみモード（ブロックしないで影響範囲確認）
  ↓
フェーズ2: IT管理者グループのみ適用（自分たちで検証）
  ↓
フェーズ3: パイロットグループ（数十名）に展開
  ↓
フェーズ4: 全社展開
```

**レポートのみモードの活用が重要です。** 新しいポリシーを適用前に影響をシミュレーションできます。

## よくあるハマりポイント

- **緊急アクセスアカウント（Break Glass）を除外設定し忘れる** → 全管理者がロックアウトされる
- **テストユーザーへの適用を忘れてレポートが不正確になる**
- **MFAの登録案内を事前周知せずに強制してヘルプデスクが炎上する**

## まとめ

条件付きアクセスはゼロトラスト実現の核心機能です。「全員MFA」から始め、デバイスコンプライアンスや場所ベースの制御を段階的に加えていくのが現実的な進め方です。必ずレポートのみモードで影響確認してから本番適用しましょう。
