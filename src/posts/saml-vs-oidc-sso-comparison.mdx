## はじめに

SaaSアプリをEntra IDでSSO連携する際、「SAMLにしますか？OIDCにしますか？」という選択が発生します。なんとなくSAMLを選んだり、アプリのドキュメント通りに設定したりしていませんか？両者の違いを理解しておくと、設定トラブルの原因特定や新規連携の設計が格段にスムーズになります。

## 一言で比較

| 項目 | SAML 2.0 | OIDC（OAuth 2.0拡張） |
|------|----------|----------------------|
| 登場時期 | 2005年（古い） | 2014年（新しい） |
| データ形式 | XML | JSON（JWT） |
| 主な用途 | エンタープライズSaaS連携 | Webアプリ・モバイルアプリ |
| 実装難易度 | やや高め | 比較的シンプル |
| SPA/モバイル適性 | 低い | 高い |

## SAMLの仕組み

```
① ユーザーがSP（Salesforceなど）にアクセス
② SP → IdP（Entra ID）にリダイレクト
③ Entra IDでサインイン
④ Entra ID → SAML AssertionをブラウザのPOSTでSPに返す
⑤ SP → SAML Assertionを検証してセッション作成
```

SAMLアサーションはXML形式で、署名によって改ざんを防ぎます。

## OIDCの仕組み

```
① ユーザーがアプリにアクセス
② アプリ → Entra IDにリダイレクト（認可リクエスト）
③ Entra IDでサインイン
④ Entra ID → 認可コードをアプリに返す
⑤ アプリ → コードをトークン（IDトークン + アクセストークン）に交換
⑥ IDトークン（JWT）でユーザーを識別
```

## どちらを選ぶべきか

### SAMLを選ぶケース

- 連携先SaaSがSAMLしかサポートしていない（Salesforce, Workday等の古いエンタープライズSaaS）
- 既存のSAML連携済み環境との互換性が必要

### OIDCを選ぶケース

- 自社開発のWebアプリ・SPAへの認証実装
- 連携先がOIDC対応している（GitHub, Google Workspace等）
- モバイルアプリへの認証実装

### Entra IDのギャラリーアプリ

Entra IDのアプリギャラリーには多くのSaaSが登録されており、SAMLかOIDCかはアプリ側の仕様で決まっています。

## 設定時のよくあるミス

### SAMLの場合

- **Entity ID（識別子）の不一致**: SP側とIdP側で完全一致が必要
- **ACS URL（返信URL）のプロトコル**: HTTPとHTTPSを間違える
- **属性マッピング**: メールアドレスのクレーム名がアプリの期待値と異なる
- **署名アルゴリズム**: SHA-1を要求する古いアプリがある

### OIDCの場合

- **リダイレクトURIの登録漏れ**: Entra IDのアプリ登録に全URIを登録する必要がある
- **スコープの設定ミス**: 必要なクレームが返ってこない

## まとめ

新規開発ならOIDC、既存エンタープライズSaaSならSAMLという大まかな指針で問題ありません。重要なのは「なぜその方式なのか」を理解した上で設定することです。トラブル時の切り分けも格段に速くなります。
