## はじめに

「Microsoft 365 Copilotを導入したら、アクセスしてはいけない人事情報がCopilotの回答に含まれていた」——これはCopilotが意図的に漏らしたのではなく、**ユーザーがすでにアクセスできる情報をCopilotがまとめた**だけです。Copilot導入前にデータガバナンスを整備することが必須です。

## M365 Copilotのデータアクセスの仕組み

Copilotはユーザーのアクセス権内にあるデータのみを参照します。

```
【Copilotが参照するデータ】
・SharePointのファイル（ユーザーがアクセスできるもの）
・OneDriveのファイル
・Teamsの会話・ファイル
・Exchangeのメール・カレンダー
・ユーザー自身のアクセス権でアクセスできる全データ
```

つまり**アクセス権が広すぎると、Copilotが余計なデータを参照してしまう**わけです。

## よくある過剰共有のパターン

### SharePointの「全員共有」

```
悪い例: 重要なSharePointサイトが「全社員（Everyone except external users）」に共有
  → Copilotがその全員に対してデータを回答に含める

良い例: 必要な部門・グループのみにアクセス権を付与
```

### 古い共有リンク

「リンクを知っている全員」形式の共有リンクが放置されている状態。

### ゲストユーザーへの広すぎるアクセス

外部パートナーが必要以上のSharePointサイトやTeamsチャンネルにアクセスできる。

## 導入前に実施すべきこと

### 1. SharePoint過剰共有の棚卸し

**SharePoint Advanced Management**を使って「最も広く共有されているサイト」を特定できます。

```
確認すべき項目:
・「全員」に共有されているサイト
・外部（ゲスト）共有が有効なサイト
・期限切れの共有リンク
・オーナーが退職したサイト
```

### 2. 秘密度ラベルの適用

Purviewの秘密度ラベルを整備し、機密性の高いファイルに「社外秘」「極秘」ラベルを適用します。Copilot導入後もラベルがアクセス制御に反映されます。

### 3. 最小権限の原則の徹底

```
見直すべき設定:
・SharePointサイトのアクセス権（グループベース管理に移行）
・Teamsチャンネルのメンバー（必要な人だけ）
・OneDriveの外部共有設定（原則無効化）
```

### 4. ゲストアクセスの制限

```
Entra ID → 外部コラボレーション設定:
・ゲストがアクセスできるコンテンツを限定
・Teamsのゲスト権限の見直し
```

## Copilotのデータガバナンスツール

| ツール | 目的 |
|--------|------|
| SharePoint Advanced Management | 過剰共有の検出・制御 |
| Microsoft Purview | 機密情報の分類・保護 |
| Microsoft 365 Copilot Dashboard | Copilotの利用状況モニタリング |
| Copilot Studio | Copilotの応答範囲のカスタマイズ |

## まとめ

Copilotは強力なツールですが「既存のアクセス権の問題を拡大する鏡」でもあります。導入前の権限棚卸しはセキュリティ観点でも必須ですし、長期的には組織のデータガバナンス成熟度を高める良い機会でもあります。
