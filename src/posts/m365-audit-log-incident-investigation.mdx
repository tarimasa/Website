## はじめに

「上司から『あのファイル、誰が外部に送った？』と聞かれた」「不審なサインインがあったと連絡が来た」——こういった場面でM365の監査ログを使って素早く調査できますか？この記事では実務で使える検索のコツを解説します。

## 監査ログの場所

**Microsoft Purview（コンプライアンスポータル）** → 監査 → 検索

または: `compliance.microsoft.com` → 監査

## 監査ログで調べられること

```
【記録されるアクティビティの例】
ファイル操作:
  ・ファイルのアクセス・ダウンロード・共有
  ・SharePointへのアップロード・削除
  ・OneDriveの操作

メール:
  ・メールの送受信
  ・メールの削除・完全削除
  ・メールの転送設定変更

サインイン・管理:
  ・ユーザーのサインイン
  ・MFAのバイパス
  ・管理者ロールの変更

Teams:
  ・メッセージの投稿・削除
  ・チームメンバーの追加・削除
```

## 実務でよく使う検索シナリオ

### シナリオ1：特定ファイルの操作履歴を調べる

「このExcelファイルを誰がダウンロードしたか」

```
開始日: [インシデント発生日の2週間前]
終了日: [今日]
アクティビティ: ファイルのダウンロード
ファイル・フォルダー・サイト: [ファイル名またはURLの一部]
```

### シナリオ2：特定ユーザーの最近の操作を調べる

「退職予定の山田さんが直近で何を操作したか」

```
開始日: [1ヶ月前]
終了日: [今日]
ユーザー: taro.yamada@contoso.com
アクティビティ: （全アクティビティ）
```

### シナリオ3：不審なサインインを調べる

「海外からのサインインがあったと通知が来た」

```
開始日: [通知の日時]
終了日: [翌日]
アクティビティ: ユーザーのサインイン
ユーザー: [対象ユーザー]
```

## 検索結果の読み方

検索結果を展開すると詳細が表示されます。

```json
{
  "CreationTime": "2026-05-25T03:15:22",
  "Id": "xxx",
  "Operation": "FileDownloaded",
  "OrganizationId": "テナントID",
  "RecordType": 6,
  "UserKey": "taro@contoso.com",
  "UserType": 0,
  "Workload": "SharePoint",
  "ClientIP": "203.0.113.45",  // ← IPアドレスで場所を特定可能
  "UserId": "taro@contoso.com",
  "SourceFileName": "重要顧客リスト.xlsx",  // ← ファイル名
  "SiteUrl": "https://contoso.sharepoint.com/sites/Sales"
}
```

## PowerShellでの一括エクスポート

大量のログをCSVに出力して分析する場合。

```powershell
# Exchange Online PowerShellモジュールが必要
Connect-ExchangeOnline

# 監査ログの検索
Search-UnifiedAuditLog `
  -StartDate "2026-05-01" `
  -EndDate "2026-05-25" `
  -UserIds "taro@contoso.com" `
  -Operations FileDownloaded, FileCopied `
  -ResultSize 5000 |
  Select-Object CreationDate, UserIds, Operations, AuditData |
  Export-Csv -Path "audit_result.csv" -Encoding UTF8
```

## 監査ログの保持期間

| ライセンス | 保持期間 |
|-----------|---------|
| Microsoft 365 Business | 90日 |
| Microsoft 365 E3 | 90日 |
| Microsoft 365 E5 | 1年 |
| Audit Premium | 10年（オプション） |

**重要**: インシデントが起きてから調べても、保持期間を過ぎていたら何も分かりません。長期保持設定は事前に検討しましょう。

## まとめ

M365の監査ログは適切に使えば強力なインシデント調査ツールです。しかし保持期間を超えたログは消えるため、事前のライセンス確認と長期保持ポリシーの設定が重要です。日頃から「どんなログが取れるか」を把握しておくことで、いざというときに素早く対応できます。
