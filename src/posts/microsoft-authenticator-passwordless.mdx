## はじめに

「パスワードをなくしたい」——セキュリティ担当者の夢ですが、組織展開は簡単ではありません。技術的な設定よりも、ユーザー教育・例外処理・ヘルプデスク対応の方が難しいことが多いです。この記事では現実的な展開ステップと、よくある躓きポイントをまとめます。

## パスワードレスとは何か

パスワードを使わずに認証する方式です。

```
【Entra IDで使えるパスワードレス方式】
1. Microsoft Authenticatorアプリ（スマホ）
   → スマホの生体認証（顔・指紋）+ 番号照合
2. FIDO2セキュリティキー（YubiKey等）
   → 物理キー + 生体認証またはPIN
3. Windows Hello for Business（PC）
   → PCの生体認証（顔・指紋）またはPIN
```

## なぜパスワードレスなのか

```
パスワードの問題:
・使い回し → 他サービスからの流出で全アカウント危険
・フィッシング → 偽サイトで入力させられる
・ブルートフォース → 総当たり攻撃
・パスワード管理の手間

パスワードレスの強み:
・パスワードがないのでフィッシングで盗まれない
・デバイス紐づきなので盗んでも使えない
・ユーザーの利便性向上（覚える必要がない）
```

## 組織展開のステップ

### フェーズ1：現状把握

```
確認事項:
・スマートフォンを持っていない社員はいるか
・社内システムがパスワードレスに対応しているか（レガシーシステム問題）
・VPN・リモートデスクトップ等のパスワード依存箇所
```

### フェーズ2：Authenticatorの有効化

1. Entra ID → 認証方法 → Microsoft Authenticator を有効化
2. 対象グループを設定（パイロットグループから）
3. ユーザーへの登録案内とヘルプ資料を準備

### フェーズ3：パイロット展開

- IT部門10〜20名で先行テスト
- 問題点を洗い出してFAQを更新
- ヘルプデスクスタッフが対応方法を習得

### フェーズ4：段階的全社展開

```
週次または月次で対象グループを拡大
  → 部門ごとに説明会を実施
  → 登録期間中はパスワードとパスワードレスを両方許可
  → 一定期間後にパスワードのみの認証を制限
```

### フェーズ5：パスワード廃止（最終目標）

条件付きアクセスでパスワード認証を禁止し、パスワードレスのみを許可。

## 例外パターンの対応

### スマートフォンを持っていないユーザー

- 共有PCのみ使用する工場ラインの従業員等
- → FIDO2キーまたはWindows Hello for Businessで代替

### 機種変更時のAuthenticator移行

Authenticatorのクラウドバックアップ（iCloud / Googleアカウント）を有効にしておく。

```
移行手順:
① 旧機種でバックアップを有効化（設定 → バックアップ）
② 新機種にAuthenticatorをインストール
③ バックアップから復元
④ 各アカウントで再登録が必要な場合もある
```

### 紛失・ロックアウト時の対応

管理者が**一時アクセスパス（TAP）**を発行してログインできるようにする。

```powershell
# Graph APIでTAPを発行する例（管理者が実行）
POST https://graph.microsoft.com/v1.0/users/{userId}/authentication/temporaryAccessPassMethods
{
  "lifetimeInMinutes": 60,
  "isUsableOnce": true
}
```

## よくある躓きポイント

| 問題 | 対策 |
|------|------|
| 「Authenticatorの使い方が分からない」問い合わせ | 動画付きのヘルプページを用意 |
| 海外出張中の機種変更でロックアウト | 出張前に別の認証方法を登録しておく |
| レガシーシステムがパスワード必須 | 例外リストを作って個別対応 |
| 番号照合の通知を間違えて「承認」 | 番号照合＋追加コンテキスト（場所情報）を有効化 |

## まとめ

パスワードレス化は技術の問題より組織変更管理（チェンジマネジメント）の問題です。ユーザー教育と例外対応の準備をしっかり行い、段階的に進めることが成功の鍵です。
