## はじめに

GPOを設定したのに「なぜか反映されない」「設定を変えたら別の場所に影響が出た」——Active Directory管理者あるあるです。GPOの優先順位と継承の仕組みを正しく理解すれば、こういった謎トラブルの原因がすっきり分かるようになります。

## GPOとは

Group Policy Object（GPO）は、Active Directoryドメイン配下のWindowsコンピューターやユーザーに設定を一括配布する仕組みです。

```
【GPOで設定できるものの例】
・パスワードポリシー（複雑さ要件・有効期限）
・画面ロックのタイムアウト
・ソフトウェアの配布・制限
・IEのプロキシ設定（レガシー）
・スクリプトの実行（ログオン・ログオフ時）
・証明書の自動配布
・Windows Defender の設定
```

## GPOとOUの関係性と推奨される設計について

GPOの適用範囲はOU構造に依存するため、OU設計とGPO設計は密接に関連しています。

GPOは「どのユーザー・コンピューターに適用されるか」を**OU（Organizational Unit）との紐付け**によって制御します。  
この関係を理解するうえで重要なのは、**OUとセキュリティグループは役割がまったく異なる**という点です。

### GPOは「OU」にリンクして適用される

GPOは以下のコンテナにリンクできます。

- サイト（Site）
- ドメイン（Domain）
- **OU（Organizational Unit）**

日常的な運用では、**OUにGPOをリンクする**のが基本です。

AD上にあるセキュリティグループとの違いは以下になります。
### OUとセキュリティグループの違い

| 項目 | OU | セキュリティグループ |
|----|----|----|
| 主な用途 | 管理単位（GPO・委任） | 権限付与 |
| GPOのリンク | ✅ 可能 | ❌ 不可（直接は不可） |
| 階層構造 | ✅ あり（親子OU） | ❌ なし |
| アクセス制御 | ❌ 不向き | ✅ 得意 |
| 管理委任 | ✅ 可能 | ❌ 不可 |

### OUの設計について
　大規模・中規模のOUの設計では、ユーザー用のOUとコンピューター用のOUを分けることが推奨されています。
これはGPOの「ユーザー構成」と「コンピューター構成」を明確に分離し可読性を向上させるとともに、将来の拡張やトラブルシュートを容易にするためです。
「ユーザー構成」はADユーザーに対して適応される設定のことであり、「コンピューター構成」はドメイン参加しているコンピューターに対して設定が適応されるものになります。
```
良いOU設計の例:
Contoso.com
  ├ Computers
  │   ├ Desktops（一般PC用OU）
  │   ├ Laptops（ノートPC用OU）
  │   └ Servers（サーバー用OU）
  └ Users
      ├ Sales（営業部ユーザー用OU）
      └ IT（IT部ユーザー用OU）
```
### その他のOUの役割
　OUにはGPOを適用するだけでなく、管理の委任ができます。

```
・特定のユーザーに対して、DesktopsのOUに属するコンピューターオブジェクトを削除する権限を与える。
・特定のユーザーに対して、SalesのOUに属するユーザーのパスワードリセットを行う権限を与える。
```

## GPO適用の優先順位（LSDOU）

GPOは以下の順番で適用され、**後から適用されたものが優先**されます。

```
L: Local（ローカルコンピューターのポリシー）← 最も弱い
S: Site（ADサイトのGPO）
D: Domain（ドメインのGPO）
O: OU（組織単位のGPO）← 最も強い
   └ 子OU → さらに上書き可能
```

```
【具体例】
ドメインGPO: パスワード最低文字数 = 8文字
OUのGPO:    パスワード最低文字数 = 12文字
→ OUのGPOが勝つ（12文字が適用される）
```

## 継承・ブロック・強制適用

### 継承（デフォルト）

親OUのGPOは子OUに継承されます。

```
ドメイン
  └ 営業部OU（ドメインGPOを継承）
      └ 営業マネージャーOU（営業部OU + ドメインGPOを継承）
```

### ブロック（Block Inheritance）

子OUで親からの継承をブロックすることができます。

```
Kiosk端末OU → 継承ブロック → 親ポリシーを無視
```

### 強制適用（Enforced / No Override）

親OUのGPOに「強制適用」を設定すると、子OUでブロックしても上書きできません。

```
ドメインGPO（強制）→ 全OU・全PC・全ユーザーに必ず適用
```

## gpresultコマンドでGPO適用結果を調べる

「どのGPOが効いているか分からない」ときの強い味方が`gpresult`コマンドです。

```powershell
# HTML形式でGPO適用レポートを生成（わかりやすい）
gpresult /H C:\temp\gpresult.html

# コンピューター設定のみ確認
gpresult /scope computer /v

# 特定ユーザーのGPO適用結果
gpresult /user username /v
```

## GPOが反映されない典型的な原因

| 原因 | 確認方法 |
|------|---------|
| セキュリティフィルタリング | GPOのプロパティ → セキュリティフィルタリング |
| GPOがリンクされていない | GPMC でOUへのリンクを確認 |
| 継承がブロックされている | GPMC のブロック表示を確認 |
| WMIフィルタの不一致 | GPOのプロパティ → WMIフィルタリング |
| グループポリシーの更新待ち | `gpupdate /force` を実行 |

```powershell
# GPOを強制的に今すぐ適用
gpupdate /force
```

## まとめ

GPOのトラブルシューティングは「LSDOUの優先順位」と「継承・ブロック・強制適用の関係」を頭に入れた上で、`gpresult /H`でレポートを確認するのが最速の解決策です。GPOが「なぜ効いているのか・なぜ効いていないのか」を説明できるようになると、AD管理者としての自信が大きく上がります。
