## はじめに

GPOを設定したのに「なぜか反映されない」「設定を変えたら別の場所に影響が出た」——Active Directory管理者あるあるです。GPOの優先順位と継承の仕組みを正しく理解すれば、こういった謎トラブルの原因がすっきり分かるようになります。

## GPOとは

Group Policy Object（GPO）は、Active Directoryドメイン配下のWindowsコンピューターやユーザーに設定を一括配布する仕組みです。

```
【GPOで設定できるものの例】
・パスワードポリシー（複雑さ要件・有効期限）
・画面ロックのタイムアウト
・ソフトウェアの配布・制限
・IEのプロキシ設定（レガシー）
・スクリプトの実行（ログオン・ログオフ時）
・証明書の自動配布
・Windows Defender の設定
```

## GPO適用の優先順位（LSDOU）

GPOは以下の順番で適用され、**後から適用されたものが優先**されます。

```
L: Local（ローカルコンピューターのポリシー）← 最も弱い
S: Site（ADサイトのGPO）
D: Domain（ドメインのGPO）
O: OU（組織単位のGPO）← 最も強い
   └ 子OU → さらに上書き可能
```

```
【具体例】
ドメインGPO: パスワード最低文字数 = 8文字
OUのGPO:    パスワード最低文字数 = 12文字
→ OUのGPOが勝つ（12文字が適用される）
```

## 継承・ブロック・強制適用

### 継承（デフォルト）

親OUのGPOは子OUに継承されます。

```
ドメイン
  └ 営業部OU（ドメインGPOを継承）
      └ 営業マネージャーOU（営業部OU + ドメインGPOを継承）
```

### ブロック（Block Inheritance）

子OUで親からの継承をブロックすることができます。

```
Kiosk端末OU → 継承ブロック → 親ポリシーを無視
```

### 強制適用（Enforced / No Override）

親OUのGPOに「強制適用」を設定すると、子OUでブロックしても上書きできません。

```
ドメインGPO（強制）→ 全OU・全PC・全ユーザーに必ず適用
```

## gpresultコマンドでGPO適用結果を調べる

「どのGPOが効いているか分からない」ときの強い味方が`gpresult`コマンドです。

```powershell
# HTML形式でGPO適用レポートを生成（わかりやすい）
gpresult /H C:\temp\gpresult.html

# コンピューター設定のみ確認
gpresult /scope computer /v

# 特定ユーザーのGPO適用結果
gpresult /user username /v
```

## GPOが反映されない典型的な原因

| 原因 | 確認方法 |
|------|---------|
| セキュリティフィルタリング | GPOのプロパティ → セキュリティフィルタリング |
| GPOがリンクされていない | GPMC でOUへのリンクを確認 |
| 継承がブロックされている | GPMC のブロック表示を確認 |
| WMIフィルタの不一致 | GPOのプロパティ → WMIフィルタリング |
| グループポリシーの更新待ち | `gpupdate /force` を実行 |

```powershell
# GPOを強制的に今すぐ適用
gpupdate /force
```

## OU設計とGPO設計の関係

GPOの適用範囲はOU構造に依存するため、OU設計とGPO設計は密接に関連しています。

```
良いOU設計の例:
Contoso.com
  ├ Computers
  │   ├ Desktops（一般PC用GPO）
  │   ├ Laptops（ノートPC用GPO）
  │   └ Servers（サーバー用GPO）
  └ Users
      ├ Sales（営業部用GPO）
      └ IT（IT部用GPO）
```

## まとめ

GPOのトラブルシューティングは「LSDOUの優先順位」と「継承・ブロック・強制適用の関係」を頭に入れた上で、`gpresult /H`でレポートを確認するのが最速の解決策です。GPOが「なぜ効いているのか・なぜ効いていないのか」を説明できるようになると、AD管理者としての自信が大きく上がります。
