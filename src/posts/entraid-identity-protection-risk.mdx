## はじめに

「海外から急にログインがあったのに誰も気づかなかった」——Identity Protectionを使えば、こういった異常なサインインをリアルタイムで検知し、自動的にMFA強制やブロックを発動できます。

## Identity Protectionとは

Entra ID P2に含まれる、機械学習ベースのリスク検知機能です。

```
【検知できる異常の例】
サインインリスク（そのサインインが疑わしい）:
  ・匿名IPアドレス（Tor等）からのアクセス
  ・通常と大きく異なる場所からのアクセス
  ・マルウェア感染デバイスからのアクセス
  ・パスワードスプレー攻撃の検知

ユーザーリスク（そのユーザーアカウントが危殆化の可能性）:
  ・ダークウェブで資格情報漏洩を検知
  ・AIが侵害されたアカウントと判定
```

## リスクレベルの種類

| レベル | 意味 |
|--------|------|
| 低 | 軽度の異常（同国内の異なる都市等） |
| 中 | 中程度の異常 |
| 高 | 明らかに不審な挙動（Tor、資格情報漏洩等） |

## リスクベース条件付きアクセスの設定

### ポリシー1：サインインリスクが中以上 → MFA強制

```
対象: 全ユーザー
条件: サインインリスク = 中以上
アクセス制御: MFAを要求
```

→ 通常使わないIPからのアクセスに自動でMFAを挟む

### ポリシー2：ユーザーリスクが高 → パスワードリセット強制

```
対象: 全ユーザー
条件: ユーザーリスク = 高
アクセス制御: アクセス許可 + MFA要求 + パスワード変更を要求
```

→ ダークウェブで漏洩が検知されたら自動でパスワード変更を強制

### ポリシー3：サインインリスクが高 → ブロック（厳格な設定）

```
対象: 全ユーザー（管理者アカウントは特に重要）
条件: サインインリスク = 高
アクセス制御: ブロック
```

## リスク検知の確認場所

Entra ID ポータル → セキュリティ → Identity Protection

- **危険なサインイン**: リスクが検知されたサインインの一覧
- **危険なユーザー**: リスクが高いと判定されたユーザーの一覧

## 誤検知への対応

リスク判定は機械学習のため誤検知もあります。

### サインインリスクを手動でリセット

「これは正常なサインインだった」と確認できた場合、リスクを手動でリセットできます。

```
危険なサインイン → 対象のサインインを選択 → 「安全として確認」
```

### ユーザーリスクをリセット

```
危険なユーザー → 対象ユーザーを選択 → 「ユーザーリスクを無視する」
```

**注意**: ユーザーリスクのリセットは軽率に行わないこと。本当に侵害されている場合があるため、必ず調査の上で実施する。

## Named Locations（既知の場所）の活用

社内のIPアドレスや特定拠点のIPを「既知の場所」として登録すると、それらの場所からのアクセスは「不審でない」と判定されやすくなります。

```
Entra ID → セキュリティ → Named Locations
  → 社内IPアドレス範囲を登録
  → 「信頼できる場所としてマーク」
```

## よくあるハマりポイント

- **海外出張者が高リスク判定でブロックされる** → 出張前に事前のMFA登録と確認を
- **VPN利用がTor扱いになることがある** → VPNのIPをNamed Locationsに登録
- **リスクポリシーはSSPR（セルフサービスPWリセット）と組み合わせが必要** → リスクが高でブロックするだけだと本人が自己解決できない

## まとめ

Identity Protectionのリスクベース条件付きアクセスは、「正常なアクセスは邪魔しない・異常なアクセスは自動で止める」というゼロトラストの理想形です。まずはサインインリスク「中以上でMFA強制」から始めて、影響を見ながら段階的に強化していきましょう。
