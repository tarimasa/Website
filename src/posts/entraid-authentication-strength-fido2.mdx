## はじめに

「MFA設定してるのにアカウントが乗っ取られた」——これはリアルタイムフィッシング（Adversary-in-the-Middle: AiTM）攻撃によるものです。SMSコードやAuthenticatorのプッシュ通知は、巧妙なフィッシングサイトを経由すると突破されてしまいます。Entra IDの「認証強度」機能を使えば、フィッシング耐性のある認証方式のみを強制できます。

## リアルタイムフィッシングとは

```
【従来のフィッシング】
偽サイト → パスワードを盗む → MFAがあれば防げる

【AiTM（リアルタイムフィッシング）】
偽サイト → ユーザーをプロキシとして本物のサイトに中継
         → ユーザーがMFAを入力 → 攻撃者が傍受
         → セッションを乗っ取り
         → MFAがあっても突破される！
```

## フィッシング耐性MFAとは

フィッシングサイトでは使えない仕組みを持つ認証方式のことです。

### FIDO2セキュリティキー

物理キー（YubiKey等）はドメインに紐づいており、偽ドメインでは使用できません。

```
本物サイト（login.microsoftonline.com）→ キーが反応する
偽サイト（l0g1n.microsoftonline.com） → キーが反応しない
```

### Windows Hello for Business

デバイスの生体認証（顔・指紋）やPINで認証。秘密鍵はデバイスのTPMに保管されており取り出せません。

## 認証強度ポリシーとは

条件付きアクセスで「このポリシーにはこの認証方式しか使えない」を強制する機能です（2023年リリース）。

### 組み込みの認証強度

| 強度 | 許可される方式 |
|------|-------------|
| 多要素認証 | すべてのMFA方式（SMS含む） |
| パスワードレスMFA | Authenticator、FIDO2、Windows Hello |
| フィッシング耐性MFA | FIDO2、Windows Hello for Businessのみ |

## 設定手順

### 管理者向けフィッシング耐性MFAの強制

1. 条件付きアクセス → 新しいポリシーを作成
2. 対象: グローバル管理者・特権ロール
3. アクセス制御 → 許可 → **認証強度を要求**
4. 「フィッシング耐性MFA」を選択

```
これにより管理者は:
・FIDO2キー または Windows Hello でのみサインイン可
・SMSやAuthenticatorプッシュは使えない（より安全）
```

### FIDO2キーの登録管理

```
Entra ID → 認証方法 → FIDO2セキュリティキー
  → 有効化
  → セルフサービス登録の許可 or 管理者が一括登録
```

## 組織展開の現実的な進め方

FIDO2キーは物理デバイスのため調達コストが発生します（1本2,000〜5,000円程度）。

```
【段階的導入】
フェーズ1: 管理者アカウント（10〜20人）にFIDO2キーを配布
フェーズ2: 機密情報へのアクセス権を持つ部門に展開
フェーズ3: Windows Hello for Businessを全社展開（コスト低）
フェーズ4: 全ユーザーのパスワードレス化
```

## よくあるハマりポイント

- **FIDO2非対応ブラウザ・デバイスが残っている** → 展開前に棚卸しが必要
- **Windows Hello for Businessの前提条件**: Hybrid Azure AD参加またはEntra参加が必要
- **緊急アクセス（Break Glass）アカウントのFIDO2管理**: キーを物理的に安全な場所に保管

## まとめ

「MFAがあれば大丈夫」は過信です。フィッシング耐性MFA（FIDO2・Windows Hello）の導入を、まず管理者アカウントから始めましょう。Entra IDの認証強度ポリシーで強制できるため、設定は難しくありません。
