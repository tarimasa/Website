## はじめに

オンプレActive Directoryへの攻撃は、ランサムウェアやAPT攻撃で非常によく使われます。しかしAD自体に高度な攻撃検知機能はなく、「気づいたときには手遅れ」というケースが後を絶ちません。Microsoft Defender for Identityを使えば、AD攻撃の典型的な手法をリアルタイムで検知できます。

## よく使われるAD攻撃手法

### Pass-the-Hash（PTH）

ユーザーのパスワードハッシュを盗み、パスワードなしで認証する攻撃。

```
攻撃者 → PC上のメモリからNTLMハッシュを抽出
       → ハッシュを使って別のサーバーに認証
       → パスワードを知らなくてもアクセス可能
```

### Kerberoasting

サービスアカウントのTGSチケットを取得し、オフラインでパスワードをクラック。

```
攻撃者 → ドメインユーザーとしてKerberosチケットをリクエスト
       → 取得したチケットをオフラインでブルートフォース
       → サービスアカウントのパスワードを解読
```

### DCSync

ドメインコントローラーを模倣してパスワードデータを複製（レプリケーション）する攻撃。

### Golden Ticket

krbtgtアカウントのハッシュを使って偽のKerberosチケットを作成。

## Defender for Identityとは

オンプレのActive DirectoryにセンサーをインストールしてADのトラフィックを分析するクラウドサービスです。

```
【構成】
ADドメインコントローラー
  ↓ センサー（エージェント）をインストール
  ↓ トラフィックをキャプチャ・分析
Microsoft Defender for Identity（クラウド）
  ↓ 機械学習で異常を検知
Microsoft Defender XDR（統合ポータル）
  ↓ アラートとして表示
```

## 検知できる攻撃の例

| 攻撃手法 | 検知内容 |
|---------|---------|
| Pass-the-Hash | 異常なハッシュ認証パターン |
| Kerberoasting | 多数のサービスチケットリクエスト |
| DCSync | 不正なレプリケーションリクエスト |
| Golden Ticket | 異常なTGTの使用 |
| Lateral Movement | 通常とは異なるサーバーへのアクセス |
| Brute Force | 短時間の大量認証失敗 |

## アラートの読み方

Defender for Identityのアラートには**タイムライン形式**で攻撃の経路が表示されます。

```
【アラート例: Kerberoasting疑い】
タイムライン:
  10:23 user01@contoso.com が 50件のSPNリクエストを実行
  10:23 対象サービスアカウント: svc-sql, svc-iis, svc-backup...
  （通常は1〜2件 → 50件は異常）

推奨対応:
  1. user01のアカウントを一時ロック
  2. サービスアカウントのパスワードをリセット
  3. 侵害の経路調査（エンドポイントのログを確認）
```

## 導入手順（概要）

1. Microsoft Defender for Identity ライセンス確認（Microsoft 365 E5等に含まれる）
2. Defender for Identityポータルでワークスペース作成
3. 全DCにセンサーをインストール
4. 30分程度でベースラインの学習が始まる
5. アラートの閾値と通知先を設定

## よくあるハマりポイント

- **全DCにセンサーを入れないと検知に漏れが発生する**
- **センサーのポート要件**: DCとDefender for Identityのクラウドサービス間の通信ポートを開ける
- **ベースライン学習期間**: 導入直後は誤検知が多め → 2〜4週間で安定する

## まとめ

Defender for IdentityはADへの攻撃を可視化する強力なツールです。特にオンプレADをまだ使っている環境では、エンドポイントセキュリティと合わせて必須の対策です。Microsoft 365 E5ライセンスに含まれているので、ライセンスを持っている組織はまず有効化してみましょう。
