## はじめに

「Microsoftログイン実装したけどトークンの仕組みがよく分からない」——ドキュメント通りに動かしたものの、アクセストークンとIDトークンの違いや、バックエンドでの検証方法が曖昧なまま運用しているケースをよく見かけます。この記事でMSAL.jsを使った認証の全体像をすっきり整理します。

## 全体アーキテクチャ

```
【フロントエンド（React / Vue等）】
  MSAL.js でEntra IDにサインイン
  → IDトークン（ユーザー識別）取得
  → アクセストークン（APIアクセス用）取得
    ↓ Authorization: Bearer {accessToken}
【バックエンドAPI（Node.js / Python等）】
  アクセストークンを検証
  → ユーザーを識別してレスポンスを返す
```

## フロントエンド実装（MSAL.js）

### インストール

```bash
npm install @azure/msal-browser @azure/msal-react
```

### MSAL設定

```typescript
import { PublicClientApplication, Configuration } from "@azure/msal-browser";

const msalConfig: Configuration = {
  auth: {
    clientId: "your-client-id",
    authority: "https://login.microsoftonline.com/your-tenant-id",
    redirectUri: "http://localhost:3000",
  },
  cache: {
    cacheLocation: "sessionStorage",
    storeAuthStateInCookie: false,
  },
};

export const msalInstance = new PublicClientApplication(msalConfig);
```

### サインインボタン

```tsx
import { useMsal } from "@azure/msal-react";

export const SignInButton = () => {
  const { instance } = useMsal();

  const handleLogin = () => {
    instance.loginPopup({
      scopes: ["openid", "profile", "User.Read"],
    });
  };

  return <button onClick={handleLogin}>Microsoftでサインイン</button>;
};
```

### アクセストークンの取得とAPI呼び出し

```typescript
import { useMsal } from "@azure/msal-react";

const { instance, accounts } = useMsal();

const callApi = async () => {
  const tokenResponse = await instance.acquireTokenSilent({
    scopes: ["api://your-backend-api/access_as_user"],
    account: accounts[0],
  });

  const response = await fetch("https://your-api.com/data", {
    headers: {
      Authorization: `Bearer ${tokenResponse.accessToken}`,
    },
  });
  return response.json();
};
```

## バックエンド実装（Node.js + Express）

### トークン検証

```typescript
import jwt from "jsonwebtoken";
import jwksClient from "jwks-rsa";

const client = jwksClient({
  jwksUri: `https://login.microsoftonline.com/{tenant-id}/discovery/v2.0/keys`,
});

const verifyToken = (token: string): Promise<jwt.JwtPayload> => {
  return new Promise((resolve, reject) => {
    const decoded = jwt.decode(token, { complete: true });
    const kid = decoded?.header?.kid;

    client.getSigningKey(kid!, (err, key) => {
      if (err) return reject(err);
      const publicKey = key!.getPublicKey();
      jwt.verify(token, publicKey, { audience: "your-client-id" }, (err, payload) => {
        if (err) return reject(err);
        resolve(payload as jwt.JwtPayload);
      });
    });
  });
};
```

## IDトークンとアクセストークンの使い分け

| トークン | 用途 | 渡す先 |
|---------|------|--------|
| IDトークン | ユーザーの身元確認（フロントエンドで使用） | フロントエンドのみ |
| アクセストークン | APIへのアクセス権限 | バックエンドAPIに送る |

**重要**: アクセストークンをフロントエンドで「解読」して使うのはアンチパターンです。バックエンドが検証します。

## よくあるハマりポイント

- **acquireTokenSilentの失敗**: セッション期限切れ → `acquireTokenPopup`へのフォールバックが必要
- **CORSエラー**: Entra IDのリダイレクトURIとアプリのURIが不一致
- **audience検証エラー**: バックエンドAPIのクライアントIDがaudクレームと一致しない

## まとめ

MSAL.jsでの認証フローの核心は「IDトークンでユーザーを識別し、アクセストークンでAPIを呼ぶ」という役割分担です。バックエンドでのトークン検証を適切に実装し、セキュリティを担保した上でMicrosoftログインを活用しましょう。
