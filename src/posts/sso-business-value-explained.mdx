## はじめに

「SSOを導入したい」と提案しても「何がどう変わるの？」と聞かれて上手く説明できなかった経験はありませんか？この記事では、SSOをビジネス課題の文脈から説明できるようになるための整理をします。

## そもそも今の課題は何か

### パスワード疲れ（Password Fatigue）

現代の会社員は平均10〜20のSaaSアカウントを持っています。

```
【1日の業務ログイン】
・PCサインイン
・メール（M365）
・社内ポータル
・Salesforce
・Slack
・Zoom
・Kintone
・経費精算システム
・人事システム
・GitHub
…
```

これだけのアカウントを管理するためにパスワードを使い回したり、メモに書いたりすることがセキュリティリスクの温床になります。

### シャドーIT

IT部門が管理していないSaaSを勝手に使い始める問題。「SSOで簡単にログインできるサービスと、面倒なサービス」があると、後者が回避されてシャドーITが生まれやすくなります。

### アカウント管理コスト

退職者の各SaaSアカウントを1つずつ削除する作業。抜け漏れがあるとセキュリティリスクになります。

## SSOが解決すること

```
【SSOで変わること】
パスワード疲れ → 1つのIDとパスワード（またはパスワードレス）で全SaaSにアクセス
シャドーIT    → 全アクセスをEntra IDで把握・管理できる
アカウント管理 → Entra IDのアカウント無効化 = 全SaaSへのアクセス即時停止
```

## 経営層・非エンジニアへの説明ポイント

### コスト削減の観点

| 項目 | 削減効果 |
|------|---------|
| パスワードリセット対応 | 1件あたり数千円のヘルプデスクコスト削減 |
| 退職時のアカウント削除 | 10分が数秒に（漏れなく） |
| パスワード疲れによる生産性低下 | 1日平均数分のログイン時間削減 |

### リスク低減の観点

- パスワード使い回しによるアカウント侵害リスクの低減
- 退職者アカウントの放置リスクゼロ
- 全SaaSへのアクセスをEntra IDで一元的に可視化・制御

## SSOの「できること」と「できないこと」

| できること | できないこと |
|-----------|------------|
| 対応SaaSへのSSO | SSO非対応のレガシーシステム |
| アクセスの一元管理 | 各SaaSの詳細な操作ログ |
| 退職時の一括無効化 | SaaSへの個別の細かい権限設定 |

## 導入優先度の考え方

```
優先度高（まず連携すべきSaaS）:
・Microsoft 365（本体なので必須）
・Salesforce、Slack、GitHub などの主要SaaS
・人事・経費システム

優先度低（後回しにしていいもの）:
・利用頻度が低いニッチなツール
・SSO対応コストが高いオンプレアプリ
```

## まとめ

SSOはセキュリティの話でもありますが、「社員の生産性向上」「管理コスト削減」「リスク低減」という3つのビジネス価値を持ちます。技術の話より先にこのビジネス価値を説明すると、経営層・業務部門への提案が通りやすくなります。
