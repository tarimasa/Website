## はじめに

「GitHubのOrganization追加お願いします」「Slackのチャンネル招待してください」「Salesforceのアカウント作ってほしい」——IT部門への個別依頼が日々飛び込んできていませんか？Entra ID Governanceの**エンタイトルメント管理（Access Packages）**を使えば、こういったアクセス申請を申請→承認→自動付与→期限失効のフローに乗せることができます。

## エンタイトルメント管理とは

アクセス権の「パッケージ」を作り、ユーザーが自分で申請・承認されたら自動付与される仕組みです。

```
【フロー】
① ユーザーがMyAccess（マイアクセスポータル）でアクセスパッケージを申請
② 承認者（マネージャーや担当者）にTeams/メールで通知
③ 承認 → 指定グループに自動追加・SaaSアクセス自動付与
④ 設定した期限（例: 90日）が来たら自動失効
⑤ 期限前に「まだ必要ですか？」の確認メールを自動送信
```

## アクセスパッケージの構成

```
アクセスパッケージ: 「Salesforce 営業担当アクセス」
  含まれるリソース:
    ・セキュリティグループ: sec-salesforce-user
    ・アプリロール: Salesforce > 標準ユーザー

  ポリシー:
    ・申請可能ユーザー: 営業部のメンバー（動的グループ）
    ・承認者: 申請者のマネージャー
    ・最大アクセス期間: 1年間
    ・アクセスレビュー: 四半期ごとにマネージャーが確認
```

## マイアクセスポータル（myaccess.microsoft.com）

ユーザーが自分でアクセスパッケージを申請できるセルフサービスポータルです。

```
ユーザーがアクセスすると:
・利用可能なアクセスパッケージ一覧が表示
・申請ボタンを押して理由を入力
・承認待ち→承認→アクセス付与まで状況が見える
```

## 外部ユーザー（B2B）への活用

エンタイトルメント管理は外部パートナーへのアクセス管理にも使えます。

```
ゲスト招待 + アクセスパッケージ:
① パートナー企業に「申請URL」を共有
② 外部ユーザーが申請（Microsoftアカウント等でサインイン）
③ 承認 → Entra IDにゲスト追加 + アクセスパッケージ付与
④ 期限（プロジェクト期間）終了で自動失効・ゲスト削除
```

## 設計のポイント

### アクセスパッケージの粒度

- 細かすぎると管理が大変、粗すぎると過剰権限
- 「業務ロール」単位でパッケージを作るのが現実的
- 例: 「営業担当基本セット」「プロジェクトXXX参加セット」

### 承認者の設計

| 設定 | 向いているケース |
|------|----------------|
| 直属マネージャー | 一般的な社内申請 |
| リソースオーナー | システム・SaaSの管理者が承認 |
| 2段階承認 | 機密性の高いリソース |

## よくあるハマりポイント

- **Entra ID P2ライセンスが必要**
- **外部ユーザーのアクセスパッケージ**: External Identitiesの設定も必要
- **グループへの自動追加**がすぐに反映されないことがある（数分程度の遅延）

## まとめ

エンタイトルメント管理はIT部門への個別依頼を減らしつつ、アクセス権の可視化・自動失効・レビューを一度に解決できる機能です。まずは主要なSaaS数本分のアクセスパッケージ作成から試してみましょう。
